# Easy Dev Spec

## 显式加载规则

- 本 skill 只能由用户显式点名加载：`$easy-dev-spec`、`easy-dev-spec`，或清楚表达“加载/使用 Easy Dev Spec skill”
- 不得因为用户提供了明确需求、Markdown 技术文档、语雀技术文档或多个代码仓库路径而自动加载
- 如果用户没有显式点名，应按普通 agent 流程处理，不应用本 skill 的追问、落盘和文档生成流程

加载后，当用户已经有相对明确的需求，希望你结合提示词、Markdown 技术文档、语雀技术文档或多个代码仓库，输出可直接指导开发的实施级 dev spec 时，使用这个 skill。

## 适用场景

- 聚焦已有代码分析、需求细化、实施级开发说明
- 支持单项目与多项目协同
- 支持提示词、Markdown 文档、本地资料、语雀文档等多输入源
- 强调接口协议、字段定义、类职责、开发顺序、联调顺序的一致性
- 分析与文档撰写期间，禁止修改业务项目代码

## 默认产物

### 单项目

默认生成 1 份项目级 dev spec：

- `<current-workspace>/.easy-coding/spec/dev/<项目名>-<需求主题>-dev-spec.md`

### 多项目

默认生成：

- 每个项目各 1 份项目级 dev spec
- 1 份跨项目总控文档

落盘规则：

- 所有产出文档默认统一写到当前工作区 `/.easy-coding/spec/dev/`
- 不因用户传入多个项目路径而自动写入各项目目录
- 生成完成后，应主动提醒用户将目标文档拷贝到对应项目工作区后再引用使用

命名规则：

- 项目级文档：`<项目名>-<需求主题>-dev-spec.md`
- 跨项目总控：`<需求主题>-cross-project-dev-spec.md`

其中：

- `项目名` 优先使用仓库目录名；若用户给了更明确的项目名，优先用用户口径
- `需求主题` 优先使用用户对本次需求的明确命名；若用户未给出，则根据需求内容生成一个简洁、稳定的主题名

## 消费约束

- 这些 dev spec 默认不属于自动扫描范围
- 不要暗示系统会自动发现并消费这些文件
- 正确做法是：由用户显式引用某一份 spec 文件给后续开发流程
- 文档中应清楚标明“适用项目”“适用范围”“应由谁引用”

## 输入发现

按以下顺序发现和使用输入：

1. 用户当前提示词
2. 用户直接粘贴的技术资料
3. 本地 Markdown / 文本文档
4. 可访问的语雀技术文档
5. 一个或多个代码仓库路径
6. 各仓库内已有的 `.easy-coding/spec/` 下非 `.easy-coding/spec/dev/`下 的文档

读取规则：

- 先静默分析，不要一上来展示大段“已读信息”
- 能从代码、配置、目录结构和现有文档中确认的信息，不要反问用户
- 对 `.easy-coding/spec/` 的读取，默认优先参考非 `spec/dev/` 文档，因为这些文档更适合作为全局设计与方案输入
- 默认不要把其他需求遗留在 `spec/dev/` 下的项目级 dev spec 作为本次输入，避免不同需求之间相互污染
- 对真正影响文档正确性的冲突项、缺失项、拍板项，再向用户追问
- 若语雀文档当前无法直接读取，应明确告知，并要求用户补充可访问内容、导出的 Markdown，或关键段落摘录

## 工作流

严格按以下顺序工作：

1. 发现输入源，确认涉及几个项目
2. 对每个项目进行静默扫描，识别技术栈、目录结构、相关模块和现有 spec
3. 识别跨项目关系：
   - 调用方向
   - 协议边界
   - 共享类 / DTO / 字段定义
   - 联调依赖
4. 先输出“阅读 / 分析 / 追问计划”
5. 基于计划进行多轮追问
6. 每轮最多问 3 个问题
7. 关键疑问收敛后，再开始正式文档生成
8. 生成项目级 dev spec
9. 若涉及多项目，再生成跨项目总控文档
10. 最后输出阅读顺序、开发顺序、联调顺序和显式引用建议

不要跳过第 4-7 步。
在关键疑问没有关闭前，不要开始写正式 spec。
在分析和文档撰写期间，不得修改业务项目代码。

## 计划阶段规则

在正式追问前，先给用户一份简洁计划。计划只在对话中展示，不默认落盘。

计划至少包含：

- 将读取哪些输入
- 将查看哪些项目 / 模块
- 已发现的冲突或缺口
- 哪些问题必须向用户确认
- 第一轮准备问哪 3 个问题

计划目标是对齐方向，不是提前输出完整分析结论。

若用户对计划提出调整，应先更新计划，再进入追问。

## 追问规则

- 每轮固定只问 3 个问题
- 若是选择题，必须使用 `A/B/C/D`
- 用户可以直接回复字母
- 优先追问阻断型问题：
  - 协议边界
  - 字段定义
  - 共享类或契约归属
  - 项目责任边界
  - 开发顺序
  - 联调顺序
  - 非功能约束
- 如果某个问题已被回答，不要换个说法重复追问
- 当问题已经足以支撑实施级 spec 时，应停止追问并开始成文

## 单项目文档要求

项目级 dev spec 需要覆盖实施所需的关键信息，而不是停留在宏观概念层。

至少要覆盖：

- 需求背景与目标
- 输入依据
- 当前代码现状
- 范围与非目标
- 模块 / 目录职责
- 接口契约
- 关键类 / 对象职责
- 数据结构与状态
- 详细实施方案
- 联调依赖
- 验收标准
- 风险与回退点

固定骨架见：

- [references/dev-spec-template.md](references/dev-spec-template.md)

## 多项目总控文档要求

当需求涉及多个项目时，除项目级文档外，还要额外生成 1 份跨项目总控文档。

这份文档用于保证：

- 每个项目都知道自己该读哪份文档
- 每个项目都知道依赖谁、被谁依赖
- 协议、字段、类定义和调用链保持一致
- 开发顺序和联调顺序明确

至少要覆盖：

- 需求全景
- 项目列表与职责边界
- 跨项目协议 / 类定义 / 字段约束
- 调用链与依赖方向
- 开发顺序建议
- 联调顺序建议
- 各项目应参考的文档清单
- 全链路验收口径
- 风险与一致性检查点

固定骨架见：

- [references/cross-project-dev-spec-template.md](references/cross-project-dev-spec-template.md)

## 冲突处理规则

冲突优先级默认如下：

1. 用户当前提示词
2. 当前代码与目录现状
3. 输入技术文档
4. 历史 spec 或其他参考文档

处理原则：

- 先输出冲突摘要，再决定是否必须追问
- 若冲突会影响协议、字段、类定义、调用关系、开发顺序或联调顺序，必须先确认
- 不要静默选边，更不要把冲突直接写进正式文档

## 文档生成约束

- 分析期间不能修改业务项目代码
- 文档撰写期间不能修改业务项目代码
- 若需要引用代码证据，应基于真实文件、目录、类型、接口、调用关系
- 若仓库中已存在相关 spec，应优先增量分析，并说明本次需求与既有文档的关系
- 若当前信息不足以支撑实施级文档，不要硬写残稿，应继续追问

## 输出收尾要求

完成文档后，结果中应明确说明：

- 生成了哪些文档
- 每个项目开发应优先阅读哪份文档
- 推荐开发顺序
- 推荐联调顺序
- 哪些协议、字段、类定义必须先统一
- 若要交给后续开发流程，应显式引用哪一份文档
- 需要将哪些文档拷贝到对应项目工作区后再引用使用
